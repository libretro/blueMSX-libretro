# Deterministic (float-free) blueMSX core

This tree has been converted to be fully deterministic: the compiled core
contains **no floating-point arithmetic** and no libc `rand()` calls, so
emulation produces bit-identical results on every platform, compiler, and
C library. Verified by disassembling every core object file: zero
FP arithmetic instructions remain.

The reference points for this work were openMSX and the MiSTer MSX cores,
both of which drive their synthesis from fixed integer/ROM tables rather
than host floating point.

## How it was done

There were three categories of floating-point use, each handled differently:

### 1. Transcendental init tables — baked as integer constants

Every `sin`/`log`/`pow` table built at chip init (OPL TL/sine/envelope
curves, YM2151/YM2413/YMF262 dB and sine tables, Okazaki-core
AR/dB2Lin/PM/AM tables, YMF278 volume curve, AY-3-8910 volt tables, SCC
FIR coefficients, mixer gain curves, the CPU-speed slider curve) is now a
frozen integer table in `Src/SoundChips/DetTables*.h` and
`Src/Libretro/DetTablesEmuFreq.h`.

The values are exactly what the original double-precision code produced on
x86-64 (IEEE-754 + glibc libm), frozen forever. They are generated by
`tools/gen_tables.c`, which reproduces the original float code verbatim
(including the places that used single-precision `powf`).

### 2. Rational-arithmetic init code — exact 64-bit integer math

Tables derived from clock/sample-rate ratios (OPL `FN_TABLE`, the
YM2413/YMF262 `fn_tab` and LFO/EG increments, YM2151 frequency/detune/noise
tables, the Okazaki core's `rate_adjust` family and `tllTable`, YMF278
volume scaling, YM2151 timer increments) are computed at runtime with
exact 64-bit integer arithmetic. `tools/gen_tables.c` **verifies every
entry bit-for-bit** against the original double math (including exhaustive
sweeps: all 65 536 dphase entries, all 32 768 tll entries, every possible
YMF278 volume input) and fails the build of the tables if any differs.

All chips run at fixed rates in this port (mixer locked to 44100 Hz,
oversampling always 1, chip clocks fixed), which is what makes the baked
values valid. The affected `setSampleRate`/`OPL_initalize` functions carry
comments stating the configuration they are baked for.

### 3. Runtime per-sample float paths — rewritten in fixed point

* `SN76489.c` — the float master-clock accumulator (`DELTA_CLOCK`) is now
  an exact rational accumulator (3579545 / (16·44100)); tone-edge
  anti-alias interpolation is Q14 fixed point. This is *more* accurate
  than the float version (no drift).
* `SCC.c` — the 95-tap FIR runs on Q30 integer coefficients with a 64-bit
  accumulator (≤ 1 LSB from the double version).
* `Ymdeltat.c` — ADPCM step/volume math uses a rational frequency base
  (`freqbaseNum/Den`, exactly 1/1 in this configuration) and 64-bit
  integer products; bit-identical to the old code.
* `AudioMixer.c` — master/channel/pan gains use Q16 lookup tables; the
  VU-meter division is 64-bit integer (also fixing a former
  divide-by-zero when master volume was 0).
* `DirAsDisk.c`, `Actions.c` — `ceil(x / 2^k)` and `ceil(x/100)` replaced
  with exact integer ceilings (bit-identical).
* `MsxGunstick.c` / `MsxAsciiLaser.c` — light-gun luminance is the exact
  rational `(2989·R + 5866·G + 1145·B) / 10000`.

### Non-float determinism fixes

`rand()` (whose sequence differs between C libraries) was used in two
audio paths and has been replaced with fixed-seed 23-bit LFSRs:
`Fmopl.c` rhythm white noise and `MameVLM5030.c` unvoiced speech samples.

### Removed

* The `DoubleT` typedef is deleted from `MsxTypes.h`. Any code that tries
  to reintroduce floating point through it fails to compile — this acts
  as a tripwire for the uncompiled legacy files (`OpenMsxY8950*.cpp`,
  `MsxAudio.cpp`, `Src/Utils/ziphelper.c`), which still contain floats
  but are not part of the build.
* TinyXML's `double` attribute APIs (unused by the core) are removed.

## Remaining `double` (intentional)

`libretro.c` fills `retro_system_timing.fps` / `.sample_rate`, which the
libretro API defines as `double`. Both are computed from exact integers
(and the compiler folds them to constants), so they are deterministic;
they exist only at the frontend API boundary and never feed emulation.

## Regenerating the tables

```
cd tools && gcc -O2 -o gen_tables gen_tables.c -lm -I. && cd ..
./tools/gen_tables    # writes Src/**/DetTables*.h, verifies integer math
```

The generator prints
`All integer reimplementations verified against double originals.`
on success and reports any mismatching entry otherwise. It is a build-time
tool only; nothing floating-point ships in the core.
